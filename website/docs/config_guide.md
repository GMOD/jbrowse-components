This page is the entry point for `config.json`, the file that defines the
assemblies, tracks, default session, and plugins for a JBrowse 2 instance.

For most users, the [JBrowse CLI](/docs/cli) (`add-assembly`, `add-track`,
`set-default-session`, etc.) is enough — you rarely need to touch `config.json`
by hand. The guides below cover what to do when you do.

## Background

- [Config basics](/docs/config_guides/intro) - shape of `config.json`, how
  assemblies and tracks are organized, and how the config is loaded at runtime
- [Configuration model](/docs/developer_guides/config_model) - how config
  schemas, slots, and config callbacks work under the hood (developer-oriented,
  but useful background)

## Reference

The pages under **API reference → Configuration schema** in the sidebar
(e.g. [BamAdapter](/docs/config/bamadapter), [LinearBasicDisplay](/docs/config/linearbasicdisplay))
are autogenerated from the schema definitions in source. They list every slot a
given element accepts, with its type and default — use them as the source of
truth when hand-editing `config.json`.

## Editing without the CLI

You can edit `config.json` in any text editor. A few tips:

- The schema is strongly typed; a malformed config can produce errors that are
  not easy to diagnose, so validate with `jbrowse --version` after large
  changes or test the file in a development instance before deploying.
- For one-off transformations, a short Node script reading and rewriting the
  JSON is often easier than scripting the CLI:

  ```js
  const config = JSON.parse(fs.readFileSync('config.json', 'utf8'))
  // mutate config.tracks, config.assemblies, etc.
  fs.writeFileSync('config.json', JSON.stringify(config, null, 2))
  ```

- The [admin server](/docs/quickstart_adminserver) provides a GUI alternative.
