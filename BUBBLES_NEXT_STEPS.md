## Medium priority — would meaningfully improve UX

### Variant density heatmap at intermediate zoom

At intermediate zoom levels (too far for individual SNPs, too close for pure
structural view), there's a visual gap. A density heatmap of variant sites per
window would bridge this — showing "this region is highly variable" before you
zoom in enough to see individual SNPs.

Needs investigation: what's the right rendering approach (per-pixel binning vs
precomputed windows), and does the bubbles data already support this or would we
need a separate summary track?

### Cross-linking between synteny and VCF views

Click a colored mark in the synteny view to highlight the corresponding variant
in the VCF track, and vice versa. Both views now use the same source data so the
coordinate mapping is straightforward.

### Phasing visualization

The bubbles now carry per-haplotype allele assignments for diploid data.
Haplotype 1 vs 2 could be colored differently to show phasing patterns across
samples — e.g., distinguishing which parental chromosome carries a variant.

Needs investigation: what color scheme avoids confusion with existing base
colors (A/C/G/T)?

## Low priority — ideas that need further evaluation

### SNP budget / max feature limit for GPU

If a user enables SNP rendering with a very large region, `expandCsOps` will
expand every feature's SNPs into GPU instances. With millions of SNPs this could
overflow GPU memory. A per-region budget or progressive LOD could help.

Needs verification: does this actually happen in practice with current zoom
thresholds, or does the bpPerPx < 50 gate already prevent it?

### Dynamic vg microservice adapter

A server-side adapter that queries a running `vg` process for subgraph
extraction. Would skip preprocessing entirely but adds deployment complexity and
per-query latency. Only makes sense if there's a use case where the
preprocessing pipeline is unacceptable (e.g., interactive exploration of graphs
that change frequently).

## GfaTabix vs VCF: what does the graph view offer beyond VCF?

The `vg deconstruct` VCF is the intermediate that drives bubbles, so a fair
question is whether the GfaTabix synteny view offers value over just using a
standard multi-sample VCF viewer. Key differences:

- **Structural context:** The synteny view shows how genomes share or diverge at
  the graph segment level — inversions, large deletions, and rearrangements are
  visible as colored blocks, not just variant records. VCF is site-centric; the
  synteny view is alignment-centric.
- **All-vs-all comparison:** The synteny view renders every genome as a row with
  segment-level coloring, giving an immediate visual of population structure.
  VCF shows genotypes per sample but requires the user to mentally reconstruct
  how samples relate.
- **Base-level + structural in one view:** At zoom-out you see structural
  variation; zoom in and SNPs appear from bubbles. VCF only shows the variant
  sites themselves.
- **Where VCF wins:** Filtering (QUAL, INFO fields), allele frequency stats,
  genotype queries, integration with existing VCF tooling (bcftools, GATK).

The two views are complementary — the synteny view for spatial/structural
context, the VCF track for detailed variant annotation. Both are now
auto-generated by `gfa-to-tabix --bubbles --output-config`.
