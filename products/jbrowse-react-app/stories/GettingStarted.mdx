# @jbrowse/react-app2

This is the [Storybook](https://storybook.js.org/) for
[`@jbrowse/react-app2`](https://www.npmjs.com/package/@jbrowse/react-app2),
which embeds the **full JBrowse 2 application** â€” menu bar, view manager, drawer
widgets, and all view types â€” inside any React component tree.

## App vs Linear Genome View â€” which to use?

JBrowse ships two embeddable React components. They serve different use cases:

- **`@jbrowse/react-app2`** is the whole app. You get every view type (linear,
  circular, dotplot, synteny, spreadsheet), the global menu bar, drag-to-resize
  panels, the import wizard, and a full session model with tabs. Pick this when
  you're building a _genome browser_ that happens to live in your page.
- **[`@jbrowse/react-linear-genome-view2`](?path=/docs/getting-started--docs)**
  is just the linear track view, no app chrome. Pick this when you want a _track
  viewer_ embedded inside a larger application that owns the layout, navigation,
  and surrounding UI.

If you're not sure, start with the LGV component â€” it's lighter weight and
easier to integrate. Move to react-app when you need the app-level features.

## Configuration shape

`react-app2` accepts the same configuration format as JBrowse Web â€” this is what
makes round-tripping configs between the two trivial:

```js
const state = createViewState({
  config: {
    configuration: {
      /* global app config (theme, RPC driver, etc.) */
    },
    assemblies: [
      {
        name: 'hg38',
        sequence: {
          /* ... */
        },
      },
    ],
    tracks: [{ trackId: 'my_track', type: 'FeatureTrack' /* ... */ }],
    defaultSession: {
      views: [
        {
          type: 'LinearGenomeView',
          init: {
            assembly: 'hg38',
            loc: 'chr1:1,000,000-1,100,000',
            tracks: ['my_track'],
            tracklist: true, // open track selector on load
          },
        },
      ],
    },
  },
})
```

Unlike JBrowse Web, the embedded component does **not** automatically fetch a
`config.json` based on URL parameters. If you want that behavior, do the fetch
yourself and pass the result to `createViewState` â€” see
[Importing config.json](?path=/docs/importing-config-json-files--docs).

When hand-authoring configs and session snapshots, the autogenerated schema
references list every available field: track/assembly slots under
[docs/config](https://jbrowse.org/jb2/docs/config/) (e.g.
[FeatureTrack](https://jbrowse.org/jb2/docs/config/featuretrack/)) and view
snapshot properties under [docs/models](https://jbrowse.org/jb2/docs/models/)
(e.g.
[LinearGenomeView](https://jbrowse.org/jb2/docs/models/lineargenomeview/)).

## Theming

Configure via `configuration.theme` in the config object, following
[MUI theme palette](https://mui.com/material-ui/customization/theming/)
conventions. See the
[theming guide](https://jbrowse.org/jb2/docs/config_guides/theme/) for the full
set of options including DNA base colors:

```js
createViewState({
  config: {
    configuration: {
      theme: {
        palette: {
          mode: 'dark',
          primary: { main: '#4d9221' },
          secondary: { main: '#c51b7d' },
        },
      },
    },
    // ...
  },
})
```

## Adding tracks at runtime

```js
// add a track at runtime (from an event handler, not during render)
state.jbrowse.addTrackConf({
  trackId: 'new_track',
  type: 'FeatureTrack',
  assemblyNames: ['hg38'] /* ... */,
})
state.session.views[0]?.showTrack('new_track')
```

See the [API Reference](?path=/docs/api-reference--docs) for the full
`createViewState` options.
