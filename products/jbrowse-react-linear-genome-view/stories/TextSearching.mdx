# Text searching

JBrowse can resolve gene symbols and other feature names typed into the location
box, jumping the view to matching features. This requires a prebuilt search
index (a `trix` file generated from the track's GFF or VCF data).

There are two flavors:

- **Aggregate index** — covers multiple tracks in a single index. Best when you
  want one global search across all annotation.
- **Per-track index** — one index per track, opened only when that track is
  loaded. Best when tracks are added/removed dynamically.

## Aggregate

Pass `aggregateTextSearchAdapters` to `createViewState`. Most apps need only a
single entry, but multiple are supported.

## Building indexes

In embedded mode you usually don't have a `config.json`, so use the `--file`
form of
[`jbrowse text-index`](https://jbrowse.org/jb2/docs/cli/#jbrowse-text-index):

### Aggregate

```sh
jbrowse text-index \
  --file file1.gff.gz --fileId trackid_of_file1 \
  --file file2.vcf.gz --fileId trackid_of_file2
```

This produces a `trix/` directory containing `aggregate.ix`, `aggregate.ixx`,
and `aggregate_meta.json`. Reference it from `aggregateTextSearchAdapters` in
your `createViewState` call.

### Per-track

Same command, single file:

```sh
jbrowse text-index --file myfile.gff3.gz --fileId trackid_of_myfile
```

Produces `myfile.gff3.gz.ix`, `myfile.gff3.gz.ixx`, and
`myfile.gff3.gz_meta.json` next to the input.

The `--fileId` should match the `trackId` you'll use at runtime — that's how a
search hit knows which track to open.

### Using an existing config

If you already maintain a `config.json` for a JBrowse Web instance, you can run
`jbrowse text-index` against it directly and copy the resulting
`aggregateTextSearchAdapters` block into your embedded `createViewState` call.
