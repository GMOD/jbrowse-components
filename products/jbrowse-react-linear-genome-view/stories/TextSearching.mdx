# Text searching

## Aggregate search index

An aggregate search index covers multiple tracks, configured via
`aggregateTextSearchAdapters` (usually a single entry, but multiple are supported).

## Preparing indexes

In embedded mode, `jbrowse text-index` normally expects a config.json, but
embedded apps often don't have one. Use the `--file` argument instead.

### Preparing aggregate search indexes

If you use the `jbrowse text-index` command with the --file argument, you can
create an aggregate index. E.g.

```sh
jbrowse text-index --file file1.gff.gz --fileId trackid_of_file1 --file file2.vcf.gz --fileId track_id_of_file2
```

This outputs a `trix/` directory containing `aggregate.ix`, `aggregate.ixx`, and
`aggregate_meta.json`, which you can reference in your embedded config. The
`--fileId` should match the trackId used at runtime so the track opens
automatically when a user selects a search result.

### Preparing per-track search indexes

For a single file, use:

```
jbrowse text-index --file myfile.gff3.gz --fileId track_id_of_myfile
```

This creates `myfile.gff3.gz.ix`, `myfile.gff3.gz.ixx`, and
`myfile.gff3.gz_meta.json`. As with the aggregate case, `--fileId` should match
the trackId used at runtime.

### Using a config file

You can also run `jbrowse text-index` against a config.json from a jbrowse-web
deployment, then copy the relevant sections (e.g. `aggregateTextSearchAdapters`)
into your `createViewState` arguments.
