# Using plugins

JBrowse plugins extend the embedded view with new track types, adapters,
renderers, view types, and menu items. They are passed to `createViewState` as
the `plugins` option and integrated into the view's plugin manager at startup.

## Inline plugins

The simplest pattern: define a `Plugin` subclass in your own source and pass it
directly. The example below registers a plugin that adds a "console.log the
selected region" action to the rubber-band menu.

## Plugins from external URLs

Plugins can also be loaded at runtime from a URL, which is the model JBrowse Web
uses for community-published plugins. The host app fetches the plugin bundle and
JBrowse registers it on the fly.

<Story of={WithExternalPlugin} />

## NPM-installed plugins

Plugins published to NPM look like inline plugins from your app's perspective —
`import` and pass the class. The snippet below isn't run in storybook (we don't
ship `jbrowse-plugin-gwas` as a workspace dependency), but it is the pattern you
would use:

```js
import GWASPlugin from 'jbrowse-plugin-gwas'
import {
  createViewState,
  JBrowseLinearGenomeView,
} from '@jbrowse/react-linear-genome-view2'

const state = createViewState({
  assembly,
  tracks,
  plugins: [GWASPlugin],
  location: 'ctgA:1105..1221',
})
```

## Plugins with web workers

If you've enabled the [Web Worker RPC](?path=/docs/web-worker-rpc--docs),
plugins must be registered in **both** the main thread and the worker —
otherwise the worker won't know about plugin-defined adapters or renderers. See
[this PR](https://github.com/GMOD/jbrowse-react-linear-genome-view-cra5-demo/pull/1/files)
for a working create-react-app example.
